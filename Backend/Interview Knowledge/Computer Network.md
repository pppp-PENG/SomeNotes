##### Linux 网络协议栈

协议栈是操作系统内核中按网络分层模型组织的一系列==函数==，通过内核态连续调用实现数据封装与解析

因此网络分层是逻辑上的，实际上在应用层进行系统调用后，操作系统会调用协议栈的相关函数进行封装与解析

##### Linux 接收网络包过程

1. 网卡接收到网络包
2. DMA 写入内存（Ring Buffer）
3. ==Linux NAPI 机制==进行中断处理

##### Linux 的 NAPI 机制

NAPI（New API）通过动态切换中断与轮询优化性能：

1. 初始触发：首个数据包到达时触发硬中断，启动处理流程。
2. 轮询模式：硬中断处理函数禁用该网卡中断，改为==由软中断（ksoftirqd线程）主动轮询网卡的接收队列（Rx Ring）==，批量处理数据包
3. 退出条件：当处理完所有数据包，或达到预设的处理预算（时间或数据包数量）时，退出轮询并重新启用中断。

##### ksoftirqd 线程

每个 CPU 各配备一个 ksoftirqd 线程，负责处理软中断

首先复制 CPU 的位图（在内存中，需要处理的软中断类型的比特位为 1），根据位图优先级处理软中断

##### Linux 发送网络包过程

1. 调用 Socket 接口进入内核态
2. 申请 sk_buff 内存，然后把数据拷贝到其中，并加入发送缓冲区
3. 网络协议栈加头部
    - sk_buff 内存会预留头部空间，还有个 ==data 指针==指向头部，网络协议栈通过移动 data 指针层层添加头部
    - 为支持丢失重传，TCP 协议会传递出 ==sk_buff 的副本==，ACK 后才会释放 sk_buff 内存
4. 添加 MAC 头部后放入网卡发送队列
5. 触发软中断，网卡驱动会从发送队列读取 sk_buff 并放到 Ring Buffer，然后通过 DMA 发送
6. 发送，网卡触发硬中断释放 sk_buff 内存并清理 Ring Buffer

【发送过程中有几次内存拷贝？

​	第一次——系统调用拷贝到 sk_buff 内存；

​	第二次——TCP 副本复制；

​	第三次——TCP 或 IP 层发现 sk_buff 大于阈值时拷贝成多个小 sk_buff（分片）】

### 2. HTTP

##### ==常见状态码==

|      | x00          | x01        | x02          | x03          | x04          | x06           |
| ---- | ------------ | ---------- | ------------ | ------------ | ------------ | ------------- |
| 2xx  | 正常         | -          | -            | -            | 无 body 数据 | body 的一部分 |
| 3xx  | -            | 永久重定向 | 临时重定向   | -            | 缓存重定向   | -             |
| 4xx  | 请求报文错误 | -          | -            | 禁止访问资源 | 资源不存在   | -             |
| 5xx  | 服务器错误   | 不支持请求 | 访问后端错误 | 服务器繁忙   | -            | -             |

2xx 表示服务器成功处理客户端的请求

- `200 OK` 正常
- `204 No Content` 正常，但无 `body` 数据
- `206 Partial Content` 正常，目前返回的 `body` 数据只是资源的一部分

3xx 表示客户端请求的资源发生变动，需要重定向

- `301 Moved Permanently` 永久重定向，浏览器会自动重定向
- `302 Found` 临时重定向（资源仍在，只是需要另一个 URL 访问），浏览器会自动重定向
- `304 Not Modified` 缓存重定向，该请求的资源仍在客户端缓存中，继续用即可

4xx 表示客户端发送的报文有误

- `400 Bad Request` 请求报文有错误（未知原因）
- `403 Forbidden` 服务器禁止访问资源
- `404 Not Found `请求的资源在服务器上不存在

5xx 表示服务器处理时内部发生错误

- `500 Internal Server Error` 服务器发生错误（未知原因）
- `501 Not Implemented` 客户端请求功能还未支持
- `502 Bad Gateway` 服务器访问后端时发生错误
- `503 Service Unavailable` 服务器当前忙碌，暂时无法响应

GET 是从服务器获得指定资源；POST 是根据请求携带的数据对指定资源进行处理

GET 是安全且幂等的；POST 是不安全且不幂等的

【安全——不会破坏服务器的资源；幂等——多次相同操作结果一致】

##### HTTP 优点

- 简单：报文格式为 `header + body`，头部是 `key-value`
- 灵活和易于扩展：工作在应用层，下层可以随意变化

### 3. HTTPS

##### HTTPS 与 HTTP 的区别

- HTTP 建立连接简单快速，HTTPS 在这过程增加了 SSL/TLS 安全协议，使报文加密传输
- HTTP 默认端口是 `80`，HTTPS 默认端口是 `443`
- HTTPS 协议需要向 CA（证书授权）申请数字证书，保证服务器身份可信

##### HTTPS 所解决的问题

1. 混合加密——解决窃听风险

2. 摘要算法 + 数字签名——解决篡改风险

3. 数字证书——解决冒充风险

##### RSA 算法

1. 客户端发起握手，包括 TLS 版本号、支持的密码套件、客户端随机数
2. 服务端发起握手，包括确认的 TLS、选择的密码套件、服务端随机数、数字证书（服务端公钥）
3. 客户端校验证书后发起握手，包括通过公钥加密的新随机数
4. 服务端发起握手，确认消息

会话密钥是三个随机数计算的结果

缺陷：因为服务端的私钥不会改变，所以==不支持向前保密==

##### ECDHE 算法

1. 客户端发起握手，包括 TLS 版本号、支持的密码套件、客户端随机数
2. 服务端==产生临时公私钥==，发起握手，包括确认的 TLS、选择的密码套件、服务端随机数、数字证书
3. 客户端==校验证书、产生临时公私钥、计算共享密钥==，发起握手，包括共享密钥
4. 服务端==计算共享密钥==，发起握手，确认消息

共享密钥由本机私钥和对方公钥计算的椭圆曲线横坐标，会话密钥是两个随机数和共享密钥计算的结果

与 RSA 主要区别：RSA 第三次握手的随机数经过了公钥加密，ECDHE 发送的共享密钥无需加密

##### 数字证书

- 签名：持有者将公钥、颁发者等信息打包，计算内容的 Hash 值后使用私钥加密生成签名
- 验证：接收者将收到的内容计算 Hash 得到 H1，使用公钥解密签名得到 H2，判断 H1 与 H2 是否相等
- 证书信任链：客户端只保存根证书的公钥，若该证书不是根证书颁发的，则请求该证书机构（把该证书机构当作一个服务器，即为迭代请求直到获得可用根证书公钥解密的证书，递归获得公钥，最后获得该中间证书公钥）

##### 优化方法

1. 软硬件升级：如更新软件版本优化一些性能；加钱加 cpu 算力（HTTPS 是计算密集型）
2. 协议优化：如密钥交换算法升级为 ECDHE；升级到 ==TLS1.3==，只需 ==1RTT== 完成 TLS 握手（因为 ECDHE 无需像 RSA 先解密证书得到公钥加密第三个随机数，可以先在最后校验证书）
3. 证书优化：使用椭圆曲线（ECDSA）证书，密钥长度短；使用 OCSP 验证证书
4. 会话复用
    - Session ID：握手后==双方==缓存会话密钥并用唯一 Session ID 标识，再次连接时服务器收到标识判断未过期则直接跳过后续 TLS 握手（服务端内存压力，负载均衡导致访问服务器不同标识失效）
    - Session Ticket：握手后==客户端==缓存加密后的会话密钥 Session Ticket，再次连接时服务器收到 Ticket 解密后判断未过期则直接跳过后续 TLS 握手
    - TLS1.3 重连：实现 ==0RTT== 重连，重连时客户端会将 Ticket 和 HTTP 请求一起发送，称为 Pre-shared Key

### 4. HTTP/1.1

##### 特点

- 使用==长连接==，减少性能开销
- 支持==管道（pipeline）==传输，发送方不需要等待接收方的响应，可以继续发送下一个请求

##### 性能瓶颈

- 未压缩请求头部（`Header`），只压缩了 `Body`
- 服务器按请求顺序响应，容易造成队头阻塞（虽然管道解决了请求的队头阻塞，但未解决响应的队头阻塞）
- 请求只能从客户端开始，服务端只能被动响应

##### 优化方法

1. 通过==缓存技术==减少发送 HTTP 请求
    - 以每次的 HTTP 请求为 `key` ，其对应的响应为 `value`，标记好过期时间，存储在本地磁盘上
    - 后续发送相同请求时，若缓存没过期直接使用，若过期了则发送请求并在 `Etag` 头部带上缓存中对应的 `value` 的摘要
    - 服务器根据客户端的 `value` 摘要判断资源是否有更新，若有更新则响应新资源，若无更新则返回不含包体的 `304` 响应，减少传输延时
2. 通过==代理服务器==优化减少重定向请求次数
    - 原本：客户端向代理服务器请求源服务器资源，代理服务器转发请求到源服务器，若源服务器返回 `302`，==代理服务器又会转发给客户端 302==，客户端再次以正确的 url 向代理服务器发送请求
    - 改进：客户端向代理服务器请求源服务器资源，代理服务器转发请求到源服务器，若源服务器返回 `302`，代理服务器自己重定向请求正确 url，==获取到资源后再给客户端==
3. 通过==合并请求==减少 HTTP 请求次数
    - HTTP/1.1 默认不使用管道，因此获取多个资源时会建立多个 TCP 连接同时获取避免队头阻塞
    - 将多个小资源打包成一个大资源，这样客户端只需一个请求即可获取，只需建立一个 TCP 连接即可
4. ==延迟发送请求==
    - 当前不需要的资源先不获取，当用户需要的时候再获取相应的资源
5. 通过==压缩==减少 HTTP 响应数据大小
    - 客户端通过请求头部的 `Accept-Encoding` 字段告诉服务器所支持的压缩算法
    - 服务器选择其中一种压缩算法压缩数据，并在 `Content-Encoding` 字段中标明

### 5. HTTP/2

##### 特点

1. 头部压缩（==HPACK 算法==）
    - 静态表：对于头部字段，客户端服务端同时维护一张静态头部信息表，用 index 代替常用头部字段
    - 动态表：在静态表（index 从 0 ~ 61）的基础上从 index = 62 开始随着通信过程更新延长该表
    - Huffman 编码：对于 value，使用 HTTP/2 编写的 Huffman 编码表进行压缩
2. 二进制格式
    - HTTP/1.x 依赖换行符和空格分割字段，解析需逐字符扫描；而 HTTP/2 帧头部固定，可直接按位解析（1.x 是根据字符解析，2 是直接解析二进制数据）
3. 并发传输
    - Frame 帧：HTTP/2 将请求切分成帧（Frame），字段、数据各单独成帧，是 HTTP/2 传输的最小单位
    - Message 消息：对应一条完整的请求或响应
    - Stream 流：虚拟双向通道，每个流都有一个唯一标识符和可选的优先级信息
    - 一个 TCP 连接可以支持数百 Stream，一个 Stream 包含一个或多个 Message，==同一 Stream 的不同 Frame 必须严格有序==
4. 服务器推送
    - 客户端服务端双方都可以建立 Stream
    - 客户端建立的 Stream 必须是奇数号，服务器被客户端请求后可以通过这个流被动发送资源
    - 服务端建立的 Stream 必须是偶数号，服务器可以通过这个流==主动发送其他资源==

##### 缺陷

只解决了应用层的队头阻塞，但未解决 TCP 的队头阻塞（==一个流发生丢包，所有流都阻塞==）

### 6. HTTP/3

##### QUIC 协议

基于 UDP 协议，但能实现可靠传输

- 无队头阻塞：也可以在一个连接上并发传输多个流，但每个流相互独立，一个流的阻塞不会影响其他流
- 更快的连接建立：不同于 TCP，==QUIC 并不与 TLS 分层==，握手时会携带 TLS 的握手环节
- 连接迁移：通过连接 ID 而非四元组标记连接，即使==网络变化导致 IP 地址改变，仍可复用原连接==，消除重连的成本

##### QPACK

利用两个单向流同步双方的动态表，避免因为==请求丢包导致后续对方解码失败==阻塞请求

- QPACK Encoder Stream：以字典（Key - Value）方式传递动态表新的一行给服务端
- QPACK Decoder Stream：服务端响应，确认已同步

### 7. TCP

##### 特点

- ==面向连接==：一对一的连接，无法一对多
- ==可靠的==：无论网络链路发生什么变化，都能保证报文一定能到达接收端
- ==基于字节流==：方便分片等

##### TCP 的连接

一个 TCP 连接包括：

- Socket：由 IP 地址和端口号组成
- 序列号：用于解决乱序问题
- 窗口大小：用于流量控制

##### TCP 与 UDP 的区别

1. 连接：TCP 传输前需要三次握手建立连接；UDP 无需建立连接，即刻传输数据
2. 服务对象：TCP 为一对一连接；UDP 支持一对一、一对多、多对多的交互通信
3. 可靠性：TCP 是可靠交付数据；UDP 是尽最大努力交付，不保证可靠
4. 拥塞控制/流量控制：TCP 具有拥塞控制和流量控制，保证数据传输安全；UDP 则没有，其发送速率不受网络拥塞影响
5. 首部开销：TCP 首部大小为 [选项] + 20 字节；UDP 首部大小固定为 8 字节
6. 传输方式：TCP 是流式传输，无边界但保证有顺；UDP 是分包传输，有边界但可能丢包和乱序
7. 分片：TCP 数据大于 MSS 时在传输层分片；UDP 数据大于 MTU 时在 IP 层分片

##### 重传机制

- SYN 或 FIN 报文发送后，未接收到对应的 ACK 即会进行重传，每次重传间隔时间为上次的两倍

##### 分片机制

- 为了配合 TCP 的重传机制，需要在 TCP 层进行分片，否则发生丢包时 IP 层无法组装成完整的 TCP 报文，导致 TCP 层让 IP 层重传这个报文的多个分片
- 以太网中 MTU（最大传输单元） 为 1500B，==MTU = IP头部长度 + TCP 头部长度 + MSS==

##### 端口

TCP 协议中会有两个端口，一个是浏览器端口（随机生成，用于区分不同进程，如多个网页），另一个是 Web 服务器监听端口==（HTTP默认为 80，HTTPS 默认为 443）==

TCP 头部中的==源端口号填写本浏览器进程端口，目的端口号填写目的服务器的 Web 监听端口==

==TCP 可以和 UDP 相同绑定同一个端口==，由网络层根据 IP 头部的传输协议决定发送给 TCP 或者 UDP 调用方法

##### 三次握手

==（第三次握手是可以携带数据的）==

【为什么使用三次握手？

​	避免非重传的历史连接初始化：客户端需要在第三次握手前确认服务端的 ACK 是否为第一次握手对应的 ACK，而不是历史的第一次握手，服务端需要等待第三次握手以确认自己没响应错】

##### 连接队列

用于存储中间状态的连接请求（队列的进出并非先进先出，而是由哈希直接定位）

- 半连接队列（SYN 队列）：服务端存储所接收的第一次握手的连接
- 全连接队列（Accept 队列）：服务端存储所接收的第三次握手的连接
- SYN 攻击：短时间伪造多个 IP 发起 TCP 请求，塞满服务端的半连接队列
- 防御 SYN 攻击：增加半连接队列的长度、启用 SYN Cookies（不使用队列，通过加密算法验证客户端第二次握手的合法性）、减少重传次数（尽快清理半连接队列）

##### 源 IP 地址的选择

当系统存在多个网卡时，会根据路由表进行选择

路由表的每行包括了 ==目标 IP - 网关 IP - 子网掩码 - 网卡编号==

根据需发送的目标 IP 地址与路由表每行子网掩码进行与操作，匹配路由表该行的路由 IP，匹配成功后就使用该行的网卡编号对应的网卡 IP 作为源 IP 地址

### 9. 数据链路层与物理层

##### MAC 头部格式

<img src="C:\Users\asus\AppData\Roaming\Typora\typora-user-images\image-20250327200944750.png" alt="image-20250327200944750" style="zoom:33%;" />

在 TCP/IP 通信中，MAC 头部中的协议类型只使用：

- IP 协议：`0x0800`
- ARP 协议：`0x0806`

==【在跨网络通信中，每一跳都会改变接收方和发送方的 MAC 地址，但源 IP 和目标 IP 地址一直不会变】==

##### ARP 协议

用于获取目标设备的 MAC 地址

先查 ARP 缓存，若存在直接用，若不存在则进行==广播==等待目标回复其 MAC 地址（默认网关不一定会永久缓存）

==【因为网络层的路由表记录的设备都与本设备处于同一子网，并且先确定了 IP 地址才会找 MAC 地址，所以 ARP 协议一定发生在局域网中】==

##### 网卡

负责将数字信息转换成电信号

网卡驱动获取到数据包后，将其复制到网卡的缓存中，然后在开头加上报头和起始帧分界符、末尾加上 FCS 校验序列

##### 网关

不同子网之间的桥梁，承担协议转换、路由与安全职能

##### 交换机/路由器

交换机位于数据链路层，而路由器位于网络层

虽然现代的路由器基本包含了交换机的功能，而且交换机只能进行有线传输，但交换机的传输带宽比无线网络大很多，所以需要局域网内的高速传输时仍会使用交换机
